# CREBAIN Comprehensive Security Review

**Date:** 2026-07-10

**Mode:** Comprehensive, infrastructure-first

**Scope:** React/Tauri application, Rust backend, IPC, scene and asset loading,
ROS/rosbridge, Zenoh, Gazebo, sensor fusion, optional NCP, dependencies, CI/CD,
repository history, and security documentation.

## Outcome

No known critical or high-severity finding remains open in the reviewed product
code. The hardening batch closes the actionable input-validation, lifecycle,
transport-correlation, fail-safe control, and supply-chain gaps found during the
review. Remaining items are explicit deployment or external-evidence boundaries,
not silently accepted product behavior.

| ID | Severity | Boundary | Resolution |
|----|----------|----------|------------|

| SEC-02 | High | Scene persistence | Scene files are JSON-only and capped at 10 MiB; browser reads are preflighted; native reads are bounded and saves use same-directory atomic replacement; schema counts, IDs, references, numbers, and migrations are validated. |
| SEC-03 | High | GLB and texture loading | Only self-contained GLB 2.0 is accepted. External URIs/subresources are rejected, embedded PNG/JPEG dimensions and aggregate pixels are bounded, and per-file/aggregate/time budgets are enforced. |
| SEC-04 | High | Gazebo mutation | Names, poses, twists, XML size, and privileged XML constructs are validated. Only the fixed bundled Maverick path may construct privileged SDF; mutation calls require an explicit successful service reply. |
| SEC-05 | High | rosbridge ingress | JSON byte arrays are rejected before generic-value amplification; strings, sequences, objects, metadata, queues, services, and payload sizes are bounded. Service replies must match both request ID and service. |
| SEC-06 | High | NCP actuation | Velocity commands are exact finite m/s vectors with a magnitude ceiling; sequence, TTL, and horizon rules fail closed; the plant is rate-limited and emits final HOLD on close/drop. RPC and spike targets are validated. |
| SEC-07 | Medium | Sensor fusion | Reconfiguration and processing are serialized safely, blocking work is separated from async locks, invalid/numerically unhealthy updates fail without poisoning state, and track/config/measurement growth is bounded. |
| SEC-08 | Medium | Frontend async lifecycle | Stale detection, bridge, scene, asset, splat, Gazebo, benchmark, and camera operations are generation/identity guarded, abortable where possible, and prevented from mutating replacement state. |

## STRIDE review

| Threat | Relevant controls | Residual deployment responsibility |
|--------|-------------------|------------------------------------|
| Spoofing | Correlated rosbridge replies, NCP session/version checks, deterministic Zenoh event names | Authenticate and isolate rosbridge/Zenoh deployments; provide reviewed ACLs/certificates |
| Tampering | Strict scene/model/image/CDR/XML validation, immutable lockfiles, pinned CI actions | Verify model digests and external artifact provenance |
| Repudiation | Structured logs, explicit service failures, candidate evidence documents | Retain deployment logs according to operator policy |
| Information disclosure | No embedded secret match, bounded structured errors, no credential files | Protect ROS/Zenoh networks and PID telemetry paths |
| Denial of service | Byte/count/pixel/time/queue caps, abort/generation guards, rate limits, bounded fusion state | Validate on target hardware with representative large assets and topology |
| Elevation of privilege | Path-root/extension validation and privileged Gazebo XML default denial | Do not enable the unsafe Gazebo XML override for untrusted callers |

## OWASP and platform review

- Injection: scene paths, model paths, ROS graph names, Gazebo XML, URLs, topics,
  CDR, and IPC payloads have boundary-specific allowlists and bounds.
- Broken access control: the desktop prototype does not implement an application
  identity layer; ROS/Zenoh authentication and authorization remain deployment
  controls and are documented as release evidence.
- Cryptographic failures: no application secrets are shipped. TLS/ACL claims are
  intentionally withheld until a target topology supplies certificates/policy.
- Insecure design and unsafe deserialization: generic unbounded JSON/CDR and
  external GLTF resource paths were replaced with constrained schemas/visitors.
- Vulnerable components: Bun audit and cargo-deny pass. Zenoh compression remains
  disabled so the ignored `lz4_flex` vulnerable block-decompression path is not
  compiled.
- Logging/monitoring: structured errors and scoped logging exist, but operator
  retention and alerting are deployment-specific.
- SSRF/resource loading: asset URLs are limited to relative, HTTPS, or loopback
  sources; redirects and streamed bytes remain bounded.
- LLM/agent security: no LLM prompt, autonomous tool, or agent runtime ships in
  the product. Repository-local development skills are not bundled into CREBAIN.

## Accepted and external boundaries

1. `RUSTSEC-2026-0041` is temporarily ignored because Zenoh 1.9 constrains
   `lz4_flex` 0.10, while CREBAIN disables `transport_compression`; remove the
   exception when Zenoh upgrades.
2. `RUSTSEC-2026-0097` is temporarily ignored because the documented custom
   global-logger interaction is not used here; remove it when a fixed `rand`
   release enters the graph.
3. cargo-deny reports allowed duplicate-version warnings and the tag-based NCP
   git declarations. The committed lockfiles resolve exact revisions; dependency
   updates must continue through frozen/locked review.
4. Direct `rmw_zenoh_cpp` interoperability needs a deployed re-keying bridge.
5. NCP remains off by default and unregistered in the product command surface;
   live Engram/realm/ACL evidence is required before enabling it.
6. The Manwe five-class `[1,9,N]` model requires an explicit adapter and golden
   fixtures before it can satisfy the native `[1,84,N]`/`[1,N,84]` contract.
7. Native launch, malicious-asset decode/render smoke, real ROS/Gazebo topology,
   and target-hardware performance remain manual release-evidence gates.

## Recommendation

Merge only with the exact candidate's hosted CI, CodeQL, and supply-chain runs
green. Do not convert any documented external boundary into a readiness claim
without the corresponding deployment artifact, fixture, or manual evidence.
